### ADR-050 The Execution-Lease Clock Is Backend-Authoritative for Shared Backends

- **Status:** accepted
- **Context:** The kernel already implements execution leases, monotonic fencing tokens, renewal, and stale-running preemption (`docs/KrakenKernelSpecification.md` §5.2; `boundaries/kernel/implementations/typescript/runtime-kernel/src/lib/runtime-kernel-runs.ts`; `migrations/0005_run_liveness.sql`), and the framework renews with an asymmetric margin (`runtime-core-liveness.ts`, `renewBeforeMs` defaulting to `leaseDurationMs/2`). Durable writes are safe: `renewLease` enforces a fencing-token CAS and `run.complete`/checkpoint rejects any non-`running` run, so a preempted worker cannot checkpoint. The gap (Architecture v0.10.0 lease-clock-skew failure class) is that expiry is judged against application-supplied wall-clock time: the original worker stamps `leaseExpiresAtMs` from `host.getNow()`, `preemptExpired` compares against a caller-supplied `nowMs`, and `renewLease` uses the kernel adapter's `now()`. In a shared-backend, multi-worker deployment these are three different clocks, so a GC-paused worker can be preempted by a peer whose clock is ahead while the first worker still believes it holds the lease — enabling a duplicated non-idempotent side effect (durable state stays fenced; external effects do not).
- **Decision:** When a backend is the shared rendezvous for more than one worker, that backend's own clock is the authority for lease stamping and expiry comparison:
  1. `@tuvren/backend-postgres` stamps `lease_expires_at_ms` and evaluates expiry using server time (`now()` / `clock_timestamp()`-derived) inside the same transaction as the lease write/compare, rather than trusting an application-supplied `nowMs`. `isLeaseExpired`, `renewLease`, and `preemptExpired` resolve "now" from backend time for shared backends, and the `renewBeforeMs` asymmetry is measured in backend time on both sides so a worker relinquishes authority before the backend deems the lease preemptable.
  2. The `BackendCapability` descriptor advertises whether the backend can serve as an authoritative shared lease clock. Single-writer embedded backends (`@tuvren/backend-memory`, single-file `@tuvren/backend-sqlite`) advertise non-support and keep the in-process clock, which is correct because no cross-worker contention exists.
  3. No kernel syscall surface change: lease operations already exist; this ADR changes only the clock source within the lease operations and the capability advertisement.
- **Consequences:** Split-brain lease-expiry under clock skew is eliminated for the shared-backend topology that a SaaS host actually runs, using the natural rendezvous clock rather than client-side drift estimation. The irreducible residue — an external side effect already in flight at the instant authority is lost — is handled by ADR-052, not by the clock. `docs/KrakenKernelSpecification.md` §5.2 and `docs/KrakenFrameworkSpecification.md` "Running Lease Ownership" are amended to state that for shared backends the lease clock is backend-authoritative. Conformance for the SaaS-readiness epics exercises preemption under simulated worker-clock skew against the PostgreSQL backend.
