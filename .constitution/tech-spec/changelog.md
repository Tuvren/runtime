The framework renewal margin (`runtime-core-liveness.ts`) is measured by elapsed local monotonic delta for shared-clock deployments so it does not mix backend-time expiry with worker-time now. - v0.30.2 - Epic BF (Data Lifecycle: Reclamation + Crypto-Shredding Erasure) closed, realizing ADR-051 (a kernel reachability-reclamation primitive plus crypto-shredding erasure of host-key-encrypted untrusted-edge payloads) exactly as accepted — no ADR text or decision changed; the realization the ADR anticipated is now repository reality, for PRD CAP-P0-066/067. The kernel gained a capability-gated `maintenance.reclaim` surface backed by a structural mark-and-sweep: it marks durable state reachable from live roots (non-archived branch heads, thread roots, active-run staged work) within the Scope, unions that with everything created at or after the grace horizon (the oldest active run's `createdAtMs`, else unbounded), and sweeps only the unreachable remainder older than the horizon, so it never releases reachable state and never races recovery; cross-branch structural sharing is honored because a shared Object stays marked via any live root. All three production backends realize the backing operation and advertise the `maintenance.reclamation` `BackendCapability` bit (memory: scoped in-memory deletes; SQLite: scoped `DELETE`s; PostgreSQL: snapshot-draft mutation + re-persist), and a backend advertising non-support rejects reclamation with `TuvrenPersistenceError` code `kernel_capability_unsupported`. Crypto-shredding erasure ships as an interface-first `PayloadCodec` contract in `@tuvren/core/lifecycle` (encrypt-on-write / decrypt-on-read keyed by a host `keyRef`, typed `ErasedPayload` result) plus a default `createAesGcmPayloadCodec({ keyring })` AES-256-GCM codec (`node:crypto`); the framework applies the codec at the provider/tool/MCP/client edges, opt-in via `createTuvren({ payloadCodec })` and back-compatible (no codec → plaintext). Keys stay host-held and the kernel stores only the opaque `TVE1` ciphertext envelope (`{ keyRef, ciphertext, alg, iv, tag }`) keyed into the content-addressed store, so destroying the host key renders the payload unrecoverable while every referencing Object/turn-node/tree/branch hash stays byte-identical — erasure rewrites no lineage (ADR-001 + Secret Isolation Model preserved). The host-facing maintenance surface on `TuvrenRuntime` drives per-Scope reclamation, and the §4.17 tenant-offboarding flow destroys the scope's keys, reclaims, and drops the scope partition (memory store drop / SQLite file removal / PostgreSQL snapshot-row delete) with no other Scope affected. The `docs/KrakenKernelSpecification.md` §9.4 reclamation claims were promoted through the authority cascade from `missing-conformance-follow-up` to `authority-backed-conformance-covered`: the kernel-protocol authority packet declares `bindingSections.reclamation` and registers the runnable `kernel-reclamation.json` plan (packet 0.7.0 → 0.8.0, plan 0.1.0 → 0.2.0), the memory/SQLite/PostgreSQL conformance adapters advertise `kernel.reclamation` and answer reclaim-probe and erasure-probe operations (the adapter plays the host/edge codec role so the kernel stays codec-unaware), and refreshed per-backend compatibility evidence records all five reclaim/erasure checks passing; the §9.4 non-support `kernel_capability_unsupported` rejection row stays with the capability-gating contract (`missing-conformance-follow-up`, `KRT-AM010`, ADR-034) because the promoted plan only exercises capability-bearing backends.
