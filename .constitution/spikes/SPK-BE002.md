# Spike Report: KRT-BE002 Scope-Binding Realization Across Backends

## 1. Context & Objective
- **Triggering upstream file/section:** `.constitution/tech-spec/adrs/ADR-048-tenancy-agnostic-scope-seam-bound-at-construction.md`; ADR-049; `docs/KrakenKernelSpecification.md` §2.3
- **Target:** The concrete per-backend realization of a construction-bound Scope (memory, SQLite, PostgreSQL) that confines every read, write, and enumeration to the constructing scope **without changing the kernel syscall surface** (ADR-048). Isolation lives entirely at the substrate; the kernel's hashing and canonical-blob identity are untouched (ADR-049).

## 2. Codebase Baseline

Verified against the current repository:

- **Construction contract.** Each backend is a factory returning a `RuntimeBackend` (`@tuvren/kernel-protocol`):
  - `createMemoryBackend(options?: { now? })` — in-memory `BackendState` (a record of `Map`s: objects, turnTrees, turnNodes, schemas, threads, branches, turns, runs, stagedResults, turnTreePaths, observeAnnotations, orderedPathChunks). `transact()` clones the state, runs the work, then swaps the committed draft in.
  - `createSqliteBackend(options: { databasePath, now? })` — one `better-sqlite3` connection per file; serialized transactions; direct SQL via `sqlite-lookups.ts` / `sqlite-repositories-*.ts`. Schema is `migrations/0001..0006`. Content tables use **single-column primary keys** (`objects.hash`, `schemas.schema_id`, `turn_trees.hash`, `turn_nodes.hash`, `threads.thread_id`, `branches.branch_id`, `turns.turn_id`, `runs.run_id`) with cross-table **foreign keys** referencing those single-column keys, plus composite-key tables (`turn_tree_paths(turn_tree_hash, path)`, `staged_results(run_id, task_id)`).
  - `createPostgresBackend(options?: PostgresBackendPersistenceOptions)` — **snapshot model**: `loadPersistedStateForUpdate(tx, schemaName)` reads the whole persisted state of a PostgreSQL **schema** into the *same in-memory `BackendState` shape the memory backend uses*, the repositories mutate a draft clone, and `persistStateSnapshot(tx, schemaName, draft, now)` writes it back. The backend already isolates by `schemaName` and reuses `memory-backend-*.ts` for its repository logic.
- **Kernel/runtime wiring.** `createRuntimeKernel()` / `createTuvren()` thread the host-supplied `RuntimeBackend` through unchanged. `RuntimeBackend` / `RuntimeBackendTx` (the syscall surface) carry **no scope argument** and must not gain one. The host-facing Durable-Read Surface (`TuvrenRuntime.listThreads` et al.) takes no scope parameter because a runtime instance is constructed against one scope-bound backend.
- **Discovered constraints.**
  - Kernel must stay scope-free (ADR-048): the scope binding lives only in the **factory options**, never in `RuntimeBackend`/`RuntimeBackendTx` method signatures or the gRPC proto.
  - Content addressing must become `(scope, hash)` (ADR-049): identical content under two scopes is two independent durable objects, with **no cross-scope dedup**.
  - SQLite content tables key purely by content hash today, so two scopes storing identical content would collide on the hash PK unless the lookup is scope-partitioned.
  - The conformance contract (`boundaries/kernel/conformance/plans/kernel-scope-isolation.json`, capability `kernel.scope-isolation`, declared by KRT-BE001) probes `store.has`, `store.get`, and enumeration cross-scope; the per-backend realization must make those probes return *isolated* results.

## 3. Options & Trade-offs

### Scope identity (shared across backends)
- **Chosen:** `Scope` is a host-supplied opaque partition identity (a non-empty string), bound at backend construction via the factory options. The host owns the mapping from its tenancy model to a `Scope`; the backend realizes the isolation. A backend constructed with no scope behaves as a single implicit default scope (back-compatible with today's single-tenant usage).
- **Rejected:** threading a `scope`/`tenant` argument through the kernel syscall surface (resolved fork Q1 in ADR-048) — churns the protocol, gRPC interop, all three backends, and the about-to-freeze public surface, and pushes tenant-shaped policy into a data-only kernel.

### Memory
- **Option A — scope-keyed shared substrate (chosen).** Introduce a `MemoryScopeStore` holding `Map<scopeKey, BackendState>`. `createMemoryBackend({ scope?, store? })` resolves (lazily creates) the per-scope `BackendState` inside the store. Two backends sharing a `store` but bound to different scopes operate on different `BackendState` objects → isolated; the same scope shares state (so per-request reconstruction of a scoped runtime retains tenant state). With no `store`, the backend allocates a private single-scope substrate → today's behavior is preserved exactly.
- **Option B — key prefixing inside one `BackendState`.** Prefix every map key with the scope. Rejected: touches every key access across the repositories and the lineage/validation helpers, and conflates the per-scope `validateCommittedState` invariants across scopes.

### SQLite
- **Option A — file-per-scope (chosen).** The Scope binds to a database file: `createSqliteBackend({ databasePath, scope? })` resolves the connection to a per-scope file (the host supplies the base path; the scope selects/derives the concrete file). Each file is an independent `(scope, hash)` space — identical content under two scopes is two physical objects; no query in one scope can observe another's rows because they are different connections. **No migration, no schema change, no query rework**, so the large existing kernel conformance suite is preserved verbatim, and existing single-scope databases keep working unchanged (the existing file is simply the default scope's file). ADR-049 explicitly sanctions "a scope-per-file ... for SQLite."
- **Option B — scope-discriminator column on a shared file.** Add a `scope` column to every content/record table, rework single-column PKs to composite `(scope, hash)` and every FK to a composite FK, scope every query, and ship a backfilling migration that assigns existing rows a default scope. Rejected for this epic: it reworks ~12 tables, every FK, and every statement in `sqlite-lookups.ts` / `sqlite-repositories-*.ts`, with high risk of regressing the extensive committed SQLite conformance evidence, for no isolation guarantee beyond what file-per-scope already gives. (Recorded as a future option if a host needs many scopes in one file.)

### PostgreSQL
- **Option A — scope-discriminator column under the snapshot model (chosen).** Keep one PostgreSQL schema; add a `scope` discriminator column to the persisted tables; `loadPersistedStateForUpdate(tx, schemaName, scope)` loads only the constructing scope's rows into the in-memory `BackendState`, and `persistStateSnapshot` writes back only that scope's rows tagged with the scope. The host supplies the scope at construction (`createPostgresBackend({ ..., scope? })`); two backends sharing the database (same schema/tables) but bound to different scopes never observe each other's rows. This realizes BE005's "two scopes share the database under row-level isolation" / "the host supplies the scope discriminator at construction, not per syscall," fits the existing whole-state snapshot model with the smallest change, and keeps content identity `(scope, hash)`.
- **Option B — dedicated schema per scope (`scope → schemaName`).** Already partially available (the backend isolates by `schemaName`). Rejected as the primary because it is schema-isolation rather than the row-level isolation BE005's acceptance criteria name, and it does not exercise the shared-substrate `(scope, hash)` model the conformance probes assert. Retained as a host-topology option (a host may still point each scope at its own schema/database).
- **Option C — host-supplied RLS connection/role with a session GUC.** The purest "row-level-isolated, host-supplied connection" reading of ADR-048. Rejected as the *default test-bearing* realization because the snapshot loader reads every visible row and RLS setup (policies + roles + `SET app.current_scope`) is host infrastructure that cannot be exercised hermetically in the conformance lane; Option A gives the same row-level guarantee enforced by the backend at construction and remains compatible with a host layering RLS on top.

## 4. Execution Directives
- **Chosen Option:** Memory = scope-keyed shared substrate (`MemoryScopeStore`); SQLite = file-per-scope; PostgreSQL = scope-discriminator column under the snapshot model. All three accept a host-supplied `Scope` at construction only; the kernel syscall surface, `RuntimeBackend`/`RuntimeBackendTx`, the gRPC proto, and the Durable-Read Surface signatures are unchanged.
- **Why it fits:** Each realization matches its substrate's nature exactly as ADR-049 prescribes (scope-keyed map / scope-per-file / row-level isolation), preserves the kernel's scope-free protocol and the existing conformance evidence, keeps content identity `(scope, hash)` with no cross-scope dedup, and is hermetically testable so KRT-BE007 can promote the `kernel-scope-isolation` plan to runnable, evidence-backed conformance.
- **Implementation notes to confirm during execution:**
  - KRT-BE003 defines the shared `Scope` type and the construction-contract extension, and realizes memory; `createTuvren` / `createRuntimeKernel` keep threading the host's (now scope-bound) backend unchanged.
  - KRT-BE004 (SQLite) confirms file-per-scope path resolution and that no migration is required; existing single-scope databases must keep working.
  - KRT-BE005 (PostgreSQL) confirms the persisted table set in `postgres-backend-persistence.ts`, adds the `scope` column + scope-filtered load/persist, and ships the schema migration; verifies two same-schema scopes are row-isolated.
  - KRT-BE006 proves the Durable-Read Surface (`listThreads`, `listBranches`, state-at-TurnNode, history walk, branch messages, `store.has`/`store.get`) is scope-confined across all three backends with no cross-scope dedup.
- **Downstream Backlog Impact:** Unlocks `KRT-BE003` (contract + memory), `KRT-BE004` (SQLite file-per-scope), `KRT-BE005` (PostgreSQL row-level discriminator), and `KRT-BE006` (scope-resolved durable identity + durable-read safety). `KRT-BE007` then makes the `kernel.scope-isolation` conformance plan runnable, registered, and evidence-backed across all three backends and promotes kernel spec §2.3 to authority-backed-conformance-covered.
